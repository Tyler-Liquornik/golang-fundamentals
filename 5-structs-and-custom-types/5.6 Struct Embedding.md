**Struct Embedding** is a form of composition that has some added features. Recall we mentioned earlier, they key difference between Java classes and Go structs is that Go is not an OOP language, so Go structs use *composition*, not *inheritance*. When you think to do inheritance as you would in OOP languages, use struct embedding instead.

Consider we want a specific type of `User`, `Admin`, that inherits the behaviour of `User` but also has `Email` and `Password` fields.

`user.go`:
```go
package user

import (
	"errors"
	"fmt"
	"time"
)

type User struct {
	FirstName string
	LastName  string
	BirthDate string
	CreatedAt time.Time
}

type Admin struct {
	email    string
	password string
	User
}

func New(firstName, lastName, birthDate string) (*User, error) {
	if (firstName == "") || (lastName == "") || (birthDate == "") {
		return nil, errors.New("first name, last name, and birth date are required")
	}

	return &User{
		FirstName: firstName,
		LastName:  lastName,
		BirthDate: birthDate,
		CreatedAt: time.Now(),
	}, nil
}

func NewAdmin(email, password string) Admin {
	return Admin{
		email,
		password,
		User{
			FirstName: "Admin",
			LastName:  "Admin",
			BirthDate: "01/01/1970",
			CreatedAt: time.Now(),
		},
	}
}

func (u User) OutputUser() {
	fmt.Println(u.FirstName, u.LastName, u.BirthDate)
}

```

What makes this pattern *embedding* and not plain *composition* is that we left the composed field to be *anonymous*. We can omit the name and it is assumed the name is the same as the type. `type Admin struct { User }` is the same as `type Admin struct { User User }`. This is typically used for struct embedding, but technically you can embed value types like `Admin struct { string }` to have a field `admin.string` although in practice you probably would never do this. If you instead use named struct fields, its just composition, and not embedding, and you do not get access to a very useful feature called **promotion**.

**Promotion** means that the fields and methods of the embedded fields can be accessed directly on the outer struct as if they were defined on it. For example, even though `Admin` *has a* a `User` (not *is a* `User` like with OOP inheritance), we can call `adminUser.OutputUser()` instead of `adminUser.User.OutputUser()`. Under the hood, nothing is copied or inherited, Admin still contains a User, but Go provides this shorthand to make composition feel more natural.

`structs.go`:
```go
package main  
  
import (  
    "fmt"  
  
    "github.com/Tyler-Liquornik/golang-fundamentals/5-structs-and-custom-types/5.1-5.6/user")  
  
func main() {  
    firstName := getUserData("Please enter your first name: ")  
    lastName := getUserData("Please enter your last name: ")  
    birthDate := getUserData("Please enter your birth date (MM/DD/YYYY): ")  
  
    appUser, err := user.New(firstName, lastName, birthDate)  
    if err != nil {  
       fmt.Println(err)  
       return  
    }  
    adminUser := user.NewAdmin("test@example.com", "password")  
  
    appUser.OutputUser()  
    adminUser.OutputUser()  
}  
  
func getUserData(promptText string) string {  
    fmt.Print(promptText)  
    var value string  
    fmt.Scanln(&value)  
    return value  
}
```

> Promotion is only available if the field is embedded (i.e., anonymous), there are **no naming conflicts** (i.e. multiple child structs expose the same field names), and the fields or methods are **accessible** (i.e., exported if used across packages).
