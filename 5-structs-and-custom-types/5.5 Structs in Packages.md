Consider we want to move the `user` struct to its own `user` package as we anticipate writing a lot more code related to it and want to separate that out. We must capitalize everything we want to expose, including the struct type, so we should now have `User` with a capital `U`.

`user.go`:
```go
package user

import (
	"errors"
	"fmt"
	"time"
)

type User struct {
	FirstName string
	LastName  string
	BirthDate string
	CreatedAt time.Time
}

func New(firstName, lastName, birthDate string) (*User, error) {
	if (firstName == "") || (lastName == "") || (birthDate == "") {
		return nil, errors.New("first name, last name, and birth date are required")
	}

	return &User{
		FirstName: firstName,
		LastName:  lastName,
		BirthDate: birthDate,
		CreatedAt: time.Now(),
	}, nil
}

func (u User) OutputUser() {
	fmt.Println(u.FirstName, u.LastName, u.BirthDate)
}
```

> The `user` fields are capitalized too, you choose to expose them individually, it is not enough to only expose the struct if we want to access individual fields (even though we don't use them directly in the example below).

`structs.go`:

```go
package main

import (
	"fmt"

	"github.com/Tyler-Liquornik/golang-fundamentals/5-structs-and-custom-types/5.1-5.6/user"
)

func main() {
	firstName := getUserData("Please enter your first name: ")
	lastName := getUserData("Please enter your last name: ")
	birthDate := getUserData("Please enter your birth date (MM/DD/YYYY): ")

	appUser, err := user.New(firstName, lastName, birthDate)
	if err != nil {
		fmt.Println(err)
		return
	}
	appUser.OutputUser()
}

func getUserData(promptText string) string {
	fmt.Print(promptText)
	var value string
	fmt.Scanln(&value)
	return value
}
```

> Often when we move a struct into a dedicated package, we even just call the constructor `new(...)`, dropping the `XXXX` from `newXXXX(...)`. We have actually already seen with `errors.New(...)`. This avoids redundancy of writing the struct name twice. since the package name will conventionally be the struct name. For an `error` struct literal you'd redundantly write `errors.NewError(...)` without this convention.
