
Earlier, when introducing pointers, we saw that value types are copied when passed into a function, so mutating them does not affect the original. This differs between Java objects and Go structs. In Java, objects the value objects pass/copy is the reference, so both the original and copy variables point to the same underlying object allowing mutations to affect the original. In Go, structs are passed by value directly, meaning the entire struct is copied, so mutations inside a function do not affect the original unless a pointer is used.

Consider we have the user in a global scope in `structs.go`, and we mutate the user passed in but then output the global struct, changing `outputUser` to `processAndOutputUser`. Then we need a pointer to process the underlying fields.

`structs.go`:
```go
package main

import (
	"fmt"
	"time"
)

type user struct {
	firstName string
	lastName  string
	birthDate string
	createdAt time.Time
}

var appUser = user{}

func main() {
	firstName := getUserData("Please enter your first name: ")
	lastName := getUserData("Please enter your last name: ")
	birthDate := getUserData("Please enter your birth date (MM/DD/YYYY): ")

	appUser.firstName = firstName
	appUser.lastName = lastName
	appUser.birthDate = birthDate
	appUser.createdAt = time.Now()

	processAndOutputUser(&appUser)
}

func processAndOutputUser(u *user) {
	u.firstName = "CHANGED"
	fmt.Println(appUser.firstName, appUser.lastName, appUser.birthDate)
}

func getUserData(promptText string) string {
	fmt.Print(promptText)
	var value string
	fmt.Scan(&value)
	return value
}
```

> Even though we passed in a pointer to a user `*user`, we don't need we can access properties of the `user` struct literal `appUser` by syntactic sugar, the compiler deals with it for us so we don't have to write like `(*u).firstName` for every property. This is different than C/C++ in this way.
