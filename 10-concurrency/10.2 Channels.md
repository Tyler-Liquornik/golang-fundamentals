The solution to our problem from the end of 10.1 is **channels**. A channel is a value used to communicate between goroutines by sending and receiving data, letting you coordinate work without directly sharing memory. You create a channel with `make(chan T)`, where `T` is the type of data it carries. By default this creates an **unbuffered channel**, meaning a send blocks until another goroutine receives (and vice versa), giving you built-in synchronization. 

`main.go`:
```go
package main

import (
	"fmt"
	"time"
)

func greet(phrase string) {
	fmt.Println("Hello!", phrase)
}

func slowGreet(phrase string, doneChan chan bool) {
	time.Sleep(3 * time.Second) // simulate a slow, long-taking task
	fmt.Println("Hello!", phrase)
	doneChan <- true
}

func main() {
	go greet("Nice to meet you!")
	go greet("How are you?")

	done := make(chan bool)
	go slowGreet("How ... are ... you ...?", done)

	go greet("I hope you're liking the course!")

	<-done
}
```

Result:
```sh
Hello! How are you?
Hello! Nice to meet you!
Hello! I hope you\'re liking the course! # Pause 3 seconds
Hello! How ... are ... you ...?
```

Now, the program runs concurrently, and also uses `<-done` to ensure main only ends once the `done` channel is closed.

> You can also use the `<-` operator to store the value the channel sends when its done.

A channel can also be used to wait for the completion of multiple goroutines. First make sure the channel is created before running any of the goroutines. Also make sure all the goroutine functions accept a channel as an argument.

Consider the code like this:

`main.go`:
```go
package main

import (
	"fmt"
	"time"
)

func greet(phrase string, doneChan chan bool) {
	fmt.Println("Hello!", phrase)
	doneChan <- true
}

func slowGreet(phrase string, doneChan chan bool) {
	time.Sleep(3 * time.Second) // simulate a slow, long-taking task
	fmt.Println("Hello!", phrase)
	doneChan <- true
}

func main() {
	done := make(chan bool)

	go greet("Nice to meet you!", done)
	go greet("How are you?", done)
	go slowGreet("How ... are ... you ...?", done)
	go greet("I hope you're liking the course!", done)

	<-done
}
```

This actually creates a race condition, because the channel just waits for one of the goroutines to finish, and doesn't care which one. Instead, the correct way to do this is to do:

`main.go`:
```go
package main

import (
	"fmt"
	"time"
)

func greet(phrase string, doneChan chan bool) {
	fmt.Println("Hello!", phrase)
	doneChan <- true
}

func slowGreet(phrase string, doneChan chan bool) {
	time.Sleep(3 * time.Second) // simulate a slow, long-taking task
	fmt.Println("Hello!", phrase)
	doneChan <- true
}

func main() {
	dones := make([]chan bool, 4)

	dones[0] = make(chan bool)
	go greet("Nice to meet you!", dones[0])
	dones[1] = make(chan bool)
	go greet("How are you?", dones[1])
	dones[2] = make(chan bool)
	go slowGreet("How ... are ... you ...?", dones[2])
	dones[3] = make(chan bool)
	go greet("I hope you're liking the course!", dones[3])

	for _, done := range dones {
		<-done
	}
}
```

Now we are waiting for 4 values to be received for `main`
 to be done. There are still race conditions, since we no longer care about the order anything finishes in, but this is ultimately the purpose of using multi-threaded code here. 
 
 However continuing to add `<-done` statements isn't very scalable, let's use another approach, iterating directly inside of the channel over the values emitted by the channel.

`main.go`:
```go
package main

import (
	"fmt"
	"time"
)

func greet(phrase string, doneChan chan bool) {
	fmt.Println("Hello!", phrase)
	doneChan <- true
}

func slowGreet(phrase string, doneChan chan bool) {
	time.Sleep(3 * time.Second) // simulate a slow, long-taking task
	fmt.Println("Hello!", phrase)
	doneChan <- true
}

func main() {
	done := make(chan bool)

	go greet("Nice to meet you!", done)
	go greet("How are you?", done)
	go slowGreet("How ... are ... you ...?", done)
	go greet("I hope you're liking the course!", done)

	for range done {}
}
```

Result:
```sh
Hello! I hope you\'re liking the course!
Hello! How are you?
Hello! Nice to meet you!
Hello! How ... are ... you ...?
fatal error: all goroutines are asleep - deadlock!

goroutine 1 [chan receive]:
main.main()
	/Users/tylerliquornik/Desktop/personal/golang-fundamentals/10-concurrency/main.go:27 +0x140
```

The issue is that Go does not know when the channel is out of values to emit, this is more like a listener. The, we are listening to all goroutines which are asleep and have no more values to emit. We can solve this by using the built-in `close` function to close the channel when its no longer needed. In our case this is at the end of `slowGreet`.

`main.go`
```go
package main

import (
	"fmt"
	"time"
)

func greet(phrase string, doneChan chan bool) {
	fmt.Println("Hello!", phrase)
	doneChan <- true
}

func slowGreet(phrase string, doneChan chan bool) {
	time.Sleep(3 * time.Second) // simulate a slow, long-taking task
	fmt.Println("Hello!", phrase)
	doneChan <- true
	close(doneChan)
}

func main() {
	done := make(chan bool)

	go greet("Nice to meet you!", done)
	go greet("How are you?", done)
	go slowGreet("How ... are ... you ...?", done)
	go greet("I hope you're liking the course!", done)

	for range done {}
}
```

Result:
```
Hello! Nice to meet you!
Hello! I hope you're liking the course!
Hello! How are you?
Hello! How ... are ... you ...?
```









You can also create a **buffered channel** with `make(chan T, N)`, where `N` is the capacity; this lets you send up to N values without blocking, acting like a small queue between goroutines.
